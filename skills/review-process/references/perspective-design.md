# 設計書レビュー観点

設計書をレビューする際に確認すべき観点の一覧。全項目を網羅的にチェックすること。

## 1. 要件との整合性

| # | チェック項目 |
|:--|:--|
| 1.1 | 対応する要件が明確に特定されているか |
| 1.2 | 要件の機能・非機能要素がすべて設計に反映されているか |
| 1.3 | 要件に存在しない機能が追加されていないか（スコープクリープ） |
| 1.4 | 要件IDとの追跡性が確保されているか |

## 2. インターフェース定義

| # | チェック項目 |
|:--|:--|
| 2.1 | 入出力パラメータが型・桁数・範囲を含めて明確に定義されているか |
| 2.2 | 外部システムとの連携仕様（API、ファイル形式等）が具体的か |
| 2.3 | 画面項目と内部データの対応関係が明確か |
| 2.4 | データフロー上の変換ルール・マッピングが定義されているか |

## 3. エラー・異常系

| # | チェック項目 |
|:--|:--|
| 3.1 | 異常系のパターンが網羅されているか（入力不正、タイムアウト、外部障害等） |
| 3.2 | 各異常パターンに対する振る舞い（エラーメッセージ、リトライ、フォールバック）が定義されているか |
| 3.3 | エラー発生時のデータ整合性（ロールバック、補償トランザクション）が考慮されているか |
| 3.4 | ユーザーへのエラー表示と内部ログの区別が明確か |

## 4. 非機能要件

| # | チェック項目 |
|:--|:--|
| 4.1 | パフォーマンス要件（応答時間、スループット）が考慮されているか |
| 4.2 | セキュリティ要件（認証、認可、暗号化、入力検証）が反映されているか |
| 4.3 | 可用性・信頼性の要件が設計に組み込まれているか |
| 4.4 | スケーラビリティの考慮がなされているか |

## 5. 用語・命名・表記

| # | チェック項目 |
|:--|:--|
| 5.1 | 用語が文書全体で統一されているか |
| 5.2 | 略語・専門用語に初出時の定義があるか |
| 5.3 | 命名規約に準拠しているか（テーブル名、カラム名、関数名等） |
| 5.4 | 表記揺れがないか（全角/半角、大文字/小文字） |

## 6. 設計の実現可能性

| # | チェック項目 |
|:--|:--|
| 6.1 | 技術的に実装可能な設計になっているか |
| 6.2 | 採用技術の制約と設計内容が矛盾していないか |
| 6.3 | 処理の実行順序に論理的な矛盾がないか |
| 6.4 | 他の設計書との整合性が取れているか（DB設計、画面設計、API設計等） |

## 7. データ設計

| # | チェック項目 |
|:--|:--|
| 7.1 | データモデルが正規化されているか（または意図的な非正規化が説明されているか） |
| 7.2 | 主キー・外部キー・ユニーク制約が適切に定義されているか |
| 7.3 | NULL許容/NOT NULL制約が設計意図に合致しているか |
| 7.4 | データ量の見積もりとインデックス戦略が考慮されているか |

## 8. 保守性・拡張性

| # | チェック項目 |
|:--|:--|
| 8.1 | 設計が変更に対して柔軟か（疎結合、関心の分離） |
| 8.2 | 将来的な要件変更に対する拡張ポイントが考慮されているか |
| 8.3 | 設定値が外部化されているか（ハードコーディングの回避） |
| 8.4 | 文書が第三者にも理解可能な粒度で記述されているか |
